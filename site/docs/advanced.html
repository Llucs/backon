Circuit Breaker PRO

The circuit breaker prevents repeated retries against a failing service, with automatic recovery. Implements three states: CLOSED (normal), OPEN (failing), HALF_OPEN (testing recovery). Thread-safe via threading.Lock.

CircuitBreaker parameters

ParameterDefaultDescription
failure_threshold5Consecutive failures before opening the circuit
recovery_timeout60.0Seconds before transitioning from OPEN to HALF_OPEN
half_open_max_calls1Allowed calls in HALF_OPEN state before fully closing
name""Identifier for the breaker
 
from backon import CircuitBreaker, BreakerRetrying, CircuitOpenError

breaker = BreakerRetrying(
    backon.expo, max_tries=3,
    breaker=CircuitBreaker(
        failure_threshold=5,
        recovery_timeout=60.0,
        half_open_max_calls=1,
    ),
)

try:
    result = breaker.call(fetch)
except CircuitOpenError:
    print("Circuit is open, skipping request")

The CircuitBreaker can also be used directly as a standalone protection for any callable:

cb = CircuitBreaker(failure_threshold=3, recovery_timeout=30.0)

# Direct call (no retry, just circuit breaking)
try:
    result = cb.call(fetch)
except CircuitOpenError:
    print("Skipping - circuit is open")

The circuit breaker tracks state transitions:

  • CLOSED — normal operation. Failures increment a counter. When failure_count >= failure_threshold, transitions to OPEN.
  • OPEN — all calls are rejected with CircuitOpenError. After recovery_timeout seconds, transitions to HALF_OPEN.
  • HALF_OPEN — allows up to half_open_max_calls probe requests. If any succeeds, transitions back to CLOSED. If any fails, transitions back to OPEN.

Hedging PRO

Run multiple retry attempts concurrently and return the first successful result. Uses ThreadPoolExecutor (sync) or asyncio tasks (async) with FIRST_COMPLETED strategy.

Hedging parameters

ParameterDefaultDescription
max_hedge3Number of concurrent hedged requests
timeoutNoneMaximum time to wait for any hedge
on_hedgeNoneCallback when a hedge request is sent
 
from backon import hedge, HedgingRetrying, on_hedge

# Functional API
result = hedge(fetch, backon.expo, max_hedge=3)

# Decorator API
@on_hedge(backon.expo, max_hedge=3)
def fetch():
    ...

# Context manager
with HedgingRetrying(backon.expo, max_hedge=3) as h:
    result = h.call(fetch)

Internally, hedging spawns max_hedge concurrent retry loops, each running _retry_sync_inner / _retry_async_inner independently. The first one to succeed returns; all others are cancelled.

Metrics PRO

Optional instrumentation with Prometheus, OpenTelemetry, and structlog. Auto-detection runs at import time (first available: prometheus_client > structlog > no-op). Each requires its corresponding package to be installed.

📈

PrometheusMetrics

Creates counters: backon_retry_attempts_total, backon_retry_success_total, backon_retry_failure_total, backon_circuit_breaker_open_total, backon_circuit_breaker_close_total, backon_hedge_requests_total.

🔭

OTelMetrics

OpenTelemetry metrics via opentelemetry-api. Same counters plus backon.retry.attempt_duration histogram. Configurable meter name.

📝

StructlogMetrics

Structured logging via structlog. Emits events: retry.attempt, retry.success, retry.failure, circuit_breaker.open, circuit_breaker.close, hedge.request.

from backon import PrometheusMetrics, OTelMetrics, StructlogMetrics, set_metrics_collector

# Prometheus
set_metrics_collector(PrometheusMetrics())

# OpenTelemetry
set_metrics_collector(OTelMetrics(meter_name="myapp.backon"))

# Structlog
set_metrics_collector(StructlogMetrics())

Query the active collector:

import backon
print(backon.get_metrics_collector())

Rate Limiter

A sliding window rate limiter that throttles retry calls to avoid overwhelming a backend. Integrated into all retry loops via the rate_limit parameter.

from backon import RateLimiter

limiter = RateLimiter(max_calls=10, period=1.0)  # max 10 calls per second

@backon.on_exception(backon.expo, ValueError, max_tries=5, rate_limit=limiter)
def fetch():
    ...
Note: RateLimitError is raised only by direct RateLimiter.__call__ / RateLimiter.acquire usage. Inside a retry loop (rate_limit=...), exceeding the limit just throttles — the loop sleeps for the computed backoff before the next attempt.

Testing Utilities

backon provides a comprehensive testing module under backon._testing (exported to the public API).

from backon import (
    disable_retries, enable_retries,
    test_config, limit_retries, remove_backoff,
    assert_retried, assert_not_retried,
)

disable_retries()

Context manager that globally disables retry within the block. Calls the target function directly.

with disable_retries():
    result = fetch()

limit_retries(n)

Context manager that overrides max retries to n for the block. Uses the internal _TEST_CONFIG dict.

with limit_retries(2):
    result = fetch()

remove_backoff()

Context manager that sets backoff multiplier to 0, eliminating all wait times. Retries happen instantly.

with remove_backoff():
    result = fetch()

test_config()

Context manager to set max_retries, backoff_multiplier, and max_backoff in a block.

assert_retried(fn, n)

Asserts that fn was retried exactly n times. Wraps the function in a counting spy and drives it through on_exception. Raises RetryAssertionError on failure.

assert_not_retried(fn)

Asserts that fn was called exactly 1 time (not retried). Same spy mechanism as assert_retried.

Trio Support

Retry with the trio async framework. The trio helpers live in backon._trio — since trio is an optional dependency, these are deliberately not part of the top-level namespace.

from backon._trio import retry_exception, retry_predicate

@retry_exception(backon.expo, ValueError, max_tries=3)
async def fetch():
    ...

Requires trio to be installed.

Retry Context Inspection

Check if code is running inside a retry and get the current attempt number anywhere in the call stack. Uses contextvars — thread-safe and async-safe.

from backon import is_retrying, get_attempt_number

def log_attempt():
    if is_retrying():
        print(f"This is attempt #{get_attempt_number()}")

@backon.on_exception(backon.expo, ValueError, max_tries=3)
def fetch():
    log_attempt()
    return api.call()

Internally, every retry loop (both fast and slow paths) wraps the target call in _retry_context_manager(attempt_number), which sets a contextvars.ContextVar for the duration of the call.

Dynamic Backoff

Override the wait time per attempt by returning a float from the giveup callback. Useful for respecting Retry-After headers.

def respect_retry_after(exc):
    return exc.response.headers.get("Retry-After", 1.0)

@backon.on_exception(backon.expo, HTTPError, giveup=respect_retry_after)
def fetch():
    ...

When giveup returns a float, backon interprets this as a custom wait time override. The retry loop uses that value as the wait duration instead of computing one from the wait generator. This allows the server to tell the client how long to wait via standard HTTP headers.

Hot Loop Detection

When 5 or more retries occur with less than 100ms between them, backon logs a warning. This helps detect misconfigured retry policies before they cause issues.

WARNING:backon:Hot loop detected: 5 retries in quick succession.
Add jitter or increase backoff.

The detection uses a global dictionary guarded by threading.Lock. It tracks timestamps of retry sleeps and resets the counter whenever a sleep interval exceeds 100ms.

Operator Composition

Compose stops, conditions, and wait generators using Python operators for a clean, expressive API.

# Stop when either condition is met
stop = stop_after_attempt(5) | stop_after_delay(30.0)

# Retry when both conditions pass
cond = retry_if_exception_type(TimeoutError) & retry_if_result(lambda x: x is None)

# Wait with combined strategy (sequential chain)
wait = backon.expo(base=3) + backon.constant(interval=0.5)

The Stop base class implements __or__ (via stop_any) and __and__ (via stop_all). RetryCondition does the same. _Wait implements __add__ (via _CombinedWait, creating a sequential chain).

Architecture

backon is organized into focused modules under the backon/ package. Each module handles a specific concern.

_retry/ package

Core retry logic split into 7 focused submodules: _api.py, _classes.py, _decide.py, _fast.py, _helpers.py, _inner.py, _loops.py. Previously a single 1277-line file.

_fast.py

Monolithic inline retry loop for the common case (no handlers, no jitter, no rate limiter). Avoids dataclass allocations and dict creation. ~3μs vs ~22μs for standard path.

_loops.py

Full-featured retry loops for sync and async, with _decide_outcome() decision logic. Handles handlers, jitter, rate limiting, attempt timeouts, and hot loop detection.

_wait_gen.py

Class-based wait generators using _Wait base class with .next() method. Replaced generator-based approach for ~15% faster iteration. Supports + operator composition.

_conditions.py

Stop and RetryCondition classes with | and & operator overloading. 20+ condition types for fine-grained retry control.

_decorator.py

on_exception and on_predicate decorators. Handles sync, async, generator, and async generator targets. Generates retry_with() method on decorated functions.

_circuit_breaker.py

CircuitBreaker with CLOSED/OPEN/HALF_OPEN states. BreakerRetrying wraps Retrying with circuit breaker protection. All state transitions are thread-safe.

_hedging.py

Hedging with ThreadPoolExecutor (sync) and asyncio tasks (async). FIRST_COMPLETED strategy with early cancellation of slower hedges.

_instrumentation.py

MetricsCollector base class, PrometheusMetrics, OTelMetrics, StructlogMetrics. Auto-detection at import time. All collectors are optional with zero hard dependencies.

_state.py

Dataclasses for retry state: RetryState, RetryCallState, Attempt, AttemptResult. Exception classes: TryAgain, RetryError, AttemptTimeoutError.

_context.py

contextvars-based retry context for is_retrying() and get_attempt_number(). Set via _retry_context_manager context manager in every retry loop.

_testing.py

Testing utilities: disable_retries, enable_retries, test_config, limit_retries, remove_backoff, assert_retried, assert_not_retried.

Fast Path Auto-Detection

The _is_fast_path() function checks if any handlers, jitter, rate limiter, attempt timeout, or state holder are configured. If none are present, the _retry_fast_sync / _retry_fast_async monolithic loops are used, avoiding all dataclass and dict allocations.

Migration from backoff

backon is a near-drop-in replacement for backoff. Change your imports and most code works as-is.

# backon import
import backon
